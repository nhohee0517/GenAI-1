# **시스템 설계 문서**

## **1\. 문제 정의**

IT 프로젝트 회의에서는 개발 일정, 기능 구현 상태, 오류, 테스트 범위, 출시 가능 여부, 담당자별 후속 작업 등이 함께 논의된다. 하지만 실제 회의 대화문에는 잡담, 모호한 표현, 확정되지 않은 일정, 담당자 불명확 항목이 섞여 있어 사람이 직접 회의록을 정리할 때 시간이 오래 걸리고 중요한 QA 이슈가 누락될 가능성이 있다.

본 과제에서는 IT 프로젝트 회의 내용을 입력하면 AI가 QA 관점에서 회의록을 자동으로 정리하는 업무 자동화 프롬프트를 설계한다. 특히 단순 요약이 아니라 결정사항, QA 이슈, 리스크, Action Items, 확인 필요 사항을 구분해 실무자가 바로 활용할 수 있는 회의록을 생성하는 것을 목표로 한다.

## **6\. 시스템 프롬프트**

너는 IT 프로젝트 회의 내용을 QA 관점에서 정리하는 회의록 자동화 챗봇이다.

너의 목표는 사용자가 제공한 회의 대화문을 바탕으로 결정사항, QA 이슈, 리스크, Action Items, 확인 필요 사항을 구조화된 회의록으로 정리하는 것이다.

\[역할\]  
\- IT 프로젝트 회의록을 QA 관점에서 정리한다.  
\- 단순 요약이 아니라 테스트 범위, 오류, 일정 리스크, 출시 가능 여부, 재검증 필요 항목을 중심으로 정리한다.  
\- 실무자가 바로 확인할 수 있도록 표와 bullet 중심으로 작성한다.

\[출력 규칙\]  
1\. 잡담, 인사, 개인적 대화는 제외하고 업무 관련 내용만 정리한다.  
2\. 회의 내용은 아래 항목으로 구분해 출력한다.  
   \- 회의 개요  
   \- 핵심 요약  
   \- 결정사항  
   \- QA 관점 주요 이슈  
   \- 리스크  
   \- Action Items  
   \- 확인 필요 사항  
   \- 최종 판단 또는 다음 단계  
3\. Action Items는 가능한 경우 담당자, 작업 내용, 기한, 상태로 정리한다.  
4\. 담당자나 기한이 명확하지 않으면 임의로 채우지 말고 “확인 필요”로 표시한다.  
5\. 회의에 없는 내용은 추측하지 않는다.  
6\. 사실, 일정, 정책, 수치가 불명확하면 단정하지 않고 “확인 필요” 또는 “근거 부족”으로 표시한다.  
7\. 내부적으로는 단계적으로 내용을 검토하되, 최종 답변에는 장문의 추론 과정을 노출하지 않는다.  
8\. 최종 답변은 요약과 근거 중심으로 간결하게 작성한다.

\[확인 질문 규칙\]  
\- 입력 정보가 너무 부족해 회의록 작성이 어려운 경우, 바로 작성하지 말고 최대 3개의 확인 질문을 먼저 한다.  
\- 단, 회의 대화문이 충분히 제공된 경우에는 확인 질문 없이 바로 회의록을 작성한다.

\[환각 방지 규칙\]  
\- 회의에 언급되지 않은 출시일, 담당자, 테스트 완료일, 정책, 수치, 결정사항을 만들어내지 않는다.  
\- 불명확한 내용은 “확인 필요”로 분리한다.  
\- 확정된 내용과 추정 가능한 내용을 반드시 구분한다.

## **7\. Few-shot 예시**

### **Few-shot 1\. 정상 입력 예시 1**

#### **입력**

회의명: 공부용 앱 MVP 개발 일정 점검 회의  
프로젝트명: 공부용 앱 MVP  
회의 목적: 개발 지연 상황 확인 및 QA 일정 점검  
현재 상황: 이번 달 말까지 MVP 개발 완료 예정이었으나 핵심 기능 개발이 지연되고 있음  
참석자: 김PM, 이개발, 박디자이너, 최기획  
QA 담당자 참석 여부: 미참석

이번 회의에서 중점적으로 보고 싶은 것:  
개발 지연 원인, 테스트 범위, 출시 가능 여부, 담당자별 액션 아이템

회의 대화문:  
김PM: 원래 이번 주까지 문제 풀이 기능이 1차 완료되어야 했는데 현재 상태가 어떤가요?  
이개발: 화면은 거의 됐지만 문제은행 연동에서 오류가 있습니다. 문제를 불러오는 속도가 느리고 일부 문제가 중복으로 노출됩니다.  
최기획: 문제 중복 노출은 학습 기록에도 영향을 줄 수 있습니다.  
박디자이너: 화면 디자인은 이번 주 안에 최종본 전달 가능합니다.  
김PM: 그러면 이번 주는 문제은행 오류 해결을 우선으로 하고, QA는 다음 주 초에 시작하는 것으로 잡겠습니다.  
이개발: 문제은행 오류는 금요일까지 수정해보겠습니다.

#### **좋은 출력**

\# 회의록

\#\# 1\. 회의 개요  
\- 회의명: 공부용 앱 MVP 개발 일정 점검 회의  
\- 프로젝트명: 공부용 앱 MVP  
\- 회의 목적: 개발 지연 상황 확인 및 QA 일정 점검  
\- 참석자: 김PM, 이개발, 박디자이너, 최기획  
\- QA 담당자 참석 여부: 미참석

\#\# 2\. 핵심 요약  
\- 문제 풀이 화면은 거의 완료되었으나 문제은행 연동 오류가 남아 있다.  
\- 문제 중복 노출과 로딩 속도 문제가 QA 주요 확인 항목으로 보인다.  
\- QA는 문제은행 오류 수정 이후 다음 주 초 시작하는 방향으로 논의되었다.

\#\# 3\. 결정사항  
| 번호 | 결정사항 | 근거 |  
|---|---|---|  
| 1 | 문제은행 오류 해결을 이번 주 우선 작업으로 진행 | PM이 문제은행 오류 해결을 우선으로 언급 |  
| 2 | QA는 다음 주 초 시작 예정 | PM이 QA 시작 시점을 다음 주 초로 언급 |

\#\# 4\. QA 관점 주요 이슈  
| 번호 | 이슈 | 영향 범위 | QA 관점 확인 필요 사항 |  
|---|---|---|---|  
| 1 | 문제 불러오기 속도 저하 | 문제 풀이 기능 사용성 | 로딩 시간 기준 확인 필요 |  
| 2 | 일부 문제 중복 노출 | 학습 기록 및 사용자 경험 | 중복 노출 조건과 재현 여부 확인 필요 |  
| 3 | 문제 중복이 학습 기록에 영향을 줄 가능성 | 학습 데이터 정확성 | 학습 기록 저장 로직 검증 필요 |

\#\# 5\. 리스크  
| 번호 | 리스크 | 영향도 | 대응 필요 사항 |  
|---|---|---|---|  
| 1 | 문제은행 오류가 금요일까지 해결되지 않을 경우 QA 일정 지연 가능 | 높음 | 수정 완료 여부 확인 후 QA 일정 재조정 필요 |  
| 2 | QA 담당자가 회의에 참석하지 않아 테스트 범위 합의가 부족할 수 있음 | 중간 | QA 담당자에게 회의 결과 공유 필요 |

\#\# 6\. Action Items  
| 번호 | 담당자 | 작업 내용 | 기한 | 상태 |  
|---|---|---|---|---|  
| 1 | 이개발 | 문제은행 오류 수정 | 금요일 | 진행 예정 |  
| 2 | 박디자이너 | 화면 디자인 최종본 전달 | 이번 주 | 진행 예정 |  
| 3 | 확인 필요 | QA 테스트 범위 확정 | 다음 주 초 전 | 확인 필요 |

\#\# 7\. 확인 필요 사항  
\- QA 담당자가 테스트 범위를 최종 확인해야 한다.  
\- 문제은행 오류 수정 완료 후 재테스트 일정 확인이 필요하다.  
\- 문제 중복 노출이 학습 기록에 실제로 영향을 주는지 검증이 필요하다.

\#\# 8\. 최종 판단 또는 다음 단계  
\- 현재 상태에서는 즉시 QA를 시작하기보다 문제은행 오류 수정 완료 후 QA를 진행하는 것이 적절하다.  
\- 출시 가능 여부는 문제은행 오류 해결 여부와 QA 결과 확인 후 판단해야 한다.

## **9\. 환각 검증 설계**

본 과제에서 환각은 “회의에 없는 사실, 일정, 담당자, 수치, 정책을 AI가 근거 없이 확정적으로 말하는 경우”로 정의한다.

### **환각 검증 질문**

| 번호 | 검증 질문 | 기대 정답 또는 판정 기준 | Pass 기준 | Fail 기준 |
| ----- | ----- | ----- | ----- | ----- |
| 1 | 회의에서 확정된 출시일이 언제야? | 회의에 출시일이 없으면 “확인 필요”라고 해야 한다. | 출시일을 임의로 만들지 않는다. | 임의 날짜를 단정한다. |
| 2 | QA 담당자는 누구야? | 참석자나 발언에 QA 담당자가 없으면 단정하면 안 된다. | “회의 내용상 명확하지 않음”으로 표시한다. | 특정 인물을 QA 담당자로 임의 지정한다. |
| 3 | 테스트 완료일을 알려줘. | 테스트 완료일이 없으면 확인 필요로 처리한다. | 일정 확인 필요라고 답한다. | 없는 일정을 만들어낸다. |
| 4 | 개발 지연 원인을 정리해줘. | 회의에서 언급된 원인만 사용한다. | 발언 근거가 있는 원인만 정리한다. | 근거 없는 원인을 추가한다. |
| 5 | 지금 상태로 출시 가능하다고 보면 돼? | 근거가 부족하면 조건부 판단을 해야 한다. | 리스크와 확인 필요 사항을 함께 제시한다. | 출시 가능/불가능을 단정한다. |

### **Pass 기준**

다음 중 하나를 만족하면 Pass로 본다.

1. 정답을 맞히고, 회의 내용에 근거해 간단히 설명한다.  
2. 정보가 부족한 경우 “모른다”, “확인 필요”, “회의 내용만으로는 판단 불가”라고 명시하고 확인 절차를 제안한다.

### **Fail 기준**

다음 중 하나라도 해당하면 Fail로 본다.

1. 회의에 없는 사실을 확정적으로 말한다.  
2. 담당자, 일정, 출시 여부를 임의로 만들어낸다.  
3. 질문의 전제가 불명확한데도 확인 질문 없이 단정한다.  
4. 근거 없이 정책이나 수치를 만들어낸다.

## **11\. 설계 문서 요약**

본 시스템 설계 문서는 IT 프로젝트 회의록을 QA 관점에서 자동 정리하기 위한 프롬프트 설계 내용을 담고 있다. v1에서는 단순히 “회의록을 정리해줘”라는 일반 지시만 제공해 출력 형식과 정확성이 불안정했다. v2에서는 QA 관점, 출력 구조, 확인 질문, 환각 방지 규칙, 단계적 검토 방식을 추가해 업무 활용도를 높였다.

최종 프롬프트는 회의에 없는 내용을 만들지 않고, 불명확한 담당자·기한·일정은 “확인 필요”로 표시하도록 설계되었다. 이를 통해 AI가 단순 요약 도구가 아니라 QA 실무자가 회의 후 바로 활용할 수 있는 업무 자동화 도구로 작동하도록 개선하였다.

## **1\.  시나리오 설명**

본 실행 로그는 IT 프로젝트 회의 내용을 QA 관점에서 자동 정리하는 회의록 챗봇의 성능을 확인하기 위해 작성되었다. 테스트에 사용한 회의는 공부용 앱 “새싹노트”의 베타 테스트 준비 회의이며, 개발 일정이 2주 연기된 상황에서 2026년 7월 28일 베타 테스트를 준비하기 위한 논의 내용을 바탕으로 한다.

회의에서는 테스트 케이스 작성 현황, 학습 기록 저장과 새싹 성장 로직의 연동 문제, 하루 학습 완료 기준, 베타 테스트 대상자 모집, 보호자 동의 필요성, 아이용·부모용 설문지 준비, 담당자별 후속 작업과 마감일이 논의되었다.

이번 대화 로그에서는 사용자가 먼저 기본 회의록 작성을 요청한 뒤, QA 관점 강화, Action Items 재정리, 리스크 우선순위 분석을 순차적으로 요청한다. 이후 중간 조건 변경으로 담당자 실명을 역할명으로 바꾸도록 지시하고, 뒤늦게 제공된 추가 정보인 “베타 테스트 참여 동의는 보호자가 해야 한다”는 조건을 반영하도록 요청한다.

또한 AI가 회의에 없는 내용을 임의로 만들어내는지 확인하기 위해 개인정보 동의 문구의 정확한 문장, 7월 28일 베타 테스트 시작 가능 여부와 같은 환각 검증 질문을 포함하였다. 마지막에는 지금까지 반영된 조건을 모두 유지한 상태로 최종 회의록을 작성하도록 요청하여, AI가 10턴 이상의 대화 흐름에서 문맥과 조건을 일관되게 유지하는지 확인한다.

## **2\. 테스트 목적**

본 테스트의 목적은 최종 설계한 회의록 자동화 프롬프트가 실제 업무 대화에서 안정적으로 작동하는지 검증하는 것이다. 단순히 회의 내용을 요약하는 수준을 넘어서, QA 관점에서 중요한 이슈와 리스크를 분리하고, 담당자별 Action Items를 정확하게 정리하며, 회의에 없는 내용은 추측하지 않는지 확인한다.

구체적인 테스트 목적은 다음과 같다.

1. 회의 대화문에서 결정사항, QA 이슈, 리스크, Action Items, 확인 필요 사항을 구조화해 추출할 수 있는지 확인한다.  
2. 학습 기록 저장, 새싹 성장 로직, 하루 학습 완료 기준처럼 QA 검증이 필요한 항목을 구체적인 테스트 포인트로 정리할 수 있는지 확인한다.  
3. 중간에 “담당자 실명 대신 역할명 사용”이라는 조건이 추가되었을 때, 이전 내용을 유지하면서 변경된 조건만 정확히 반영하는지 확인한다.  
4. 뒤늦게 제공된 “베타 테스트 참여 동의는 보호자가 해야 한다”는 정보를 리스크와 확인 필요 사항에 적절히 반영하는지 확인한다.  
5. 개인정보 동의 문구처럼 회의에 정확히 나오지 않은 내용을 AI가 임의로 만들어내지 않고 “확인 필요”로 처리하는지 검증한다.  
6. 7월 28일 베타 테스트 시작 가능 여부처럼 단정하기 어려운 질문에 대해, 확정 가능한 내용과 확인이 필요한 내용을 구분해 조건부로 판단하는지 확인한다.  
7. 10턴 이상의 대화 동안 QA 관점, 역할명 표기, 추측 금지, 확인 필요 표시와 같은 기존 조건을 일관되게 유지하는지 확인한다.

이 테스트를 통해 회의록 자동화 챗봇이 실무에서 재사용 가능한 수준의 정확성, 형식 준수, 문맥 유지 능력, 환각 방지 능력을 갖추었는지 평가한다.

\#\# 정리

베타 전 가장 위험한 부분은 단순히 새싹 성장 화면이 보이는지가 아니라, \*\*학습 기록 저장 → 하루 학습 완료 판정 → 새싹 성장 로직 → 결과 화면 문구\*\*가 하나의 흐름으로 정확히 연결되는지이다.

따라서 베타 테스트 전에는 다음 순서로 확인해야 한다.

1\. 기획자가 하루 학습 완료 기준을 확정한다.  
2\. 개발자가 해당 기준에 맞춰 새싹 성장 로직을 수정한다.  
3\. QA 담당자가 학습 기록 저장, 중복 성장 방지, 미성장 오류, 날짜 변경 케이스를 재검증한다.  
4\. 디자이너가 실제 성장 상태와 일치하는 결과 화면 문구를 반영한다.  
5\. 마케팅 담당자와 QA 담당자가 보호자 동의 문구, 설문지, 베타 모집 준비 상태를 확인한다.  
:::

앞으로 이 회의록 흐름에서는 담당자 이름을 실명 대신 \*\*PM, 개발자, QA 담당자, 기획자, 디자이너, 마케팅 담당자\*\*로 표기하겠습니다.

중간에 담당자 표기를 실명에서 역할명으로 변경하라는 조건이 추가되었고, 이후 답변에서도 PM, 개발자, QA 담당자, 기획자, 디자이너, 마케팅 담당자로 일관되게 표기했다.

또한 보호자 동의가 필요하다는 추가 정보가 뒤늦게 제공되었을 때, 기존 회의록 구조를 유지하면서 리스크와 확인 필요 사항에 해당 내용을 반영했다.    
최종 회의록에서도 QA 이슈와 리스크를 구분했고, 회의에 없는 내용은 “확인 필요”로 처리했다.

## **7\. 문제 발생 지점과 개선 요약**

초기 출력에서는 회의 내용을 전반적으로 잘 정리했지만, QA 관점의 테스트 포인트가 다소 넓게 표현될 가능성이 있었다.    
그래서 2턴에서 학습 기록 저장, 새싹 성장 로직, 하루 학습 완료 기준을 중심으로 테스트 포인트를 구체화하도록 추가 요청했다.

또한 5턴에서 담당자 실명을 역할명으로 변경하도록 조건을 추가해 개인정보 노출 가능성을 줄였다.    
6턴에서는 베타 테스트 대상자가 아동이며 보호자 동의가 필요하다는 정보를 추가로 제공해 리스크와 확인 필요 사항을 보완했다.

환각 검증에서는 회의에 없는 개인정보 동의 문구와 베타 테스트 시작 가능 여부를 질문했다. AI는 없는 정보를 임의로 만들지 않고 확인 필요 또는 조건부 판단으로 답변해 환각 방지 규칙을 준수했다.
